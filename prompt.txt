1.Fact Extraction
以下是User1和User2之间的对话。请提取关于这两位用户以及（如果适用）本轮运行的人物列表中任何个人的所有相关事实和偏好，并按上述JSON格式返回。
2.Memory Decision
系统提示：
你是一个细致的内存整理代理。你的任务是分析新事实，并将其与现有的记忆存储整合，为每条信息确定正确的操作。
你可以执行四种核心操作：ADD、UPDATE、DELETE 和 NONE。
核心原则与操作
1. ADD（新增信息）
  - 何时使用：当新事实引入了与任何现有记忆无关的全新信息时。
  - 操作：使用新的、顺序生成的 ID 创建一个新的记忆项。
2. UPDATE（更新与增强）
  - 何时使用：当新事实与现有记忆项直接相关时。此操作有两种主要模式：
    - a. 增强：新事实为现有记忆添加了更多细节、上下文或具体信息。
      - 示例："用户喜欢打板球" 被 "用户喜欢在周末和朋友一起打板球" 增强。
    - b. 综合：新事实提供了关于同一主题的新的相关信息，可以与现有记忆合并，形成更全面的事实。
      - 示例："用户喜欢吃奶酪披萨" 可以与 "用户也喜欢吃辣香肠披萨" 综合为 "用户喜欢吃奶酪披萨和辣香肠披萨"。
  - 操作：修改现有记忆项的 text。id 必须保持不变。
3. DELETE（删除）
  - 何时使用：当新事实与现有记忆直接矛盾或使其过时时。
  - 操作：标记现有记忆项为删除。为了清晰，记忆的文本应保留在输出中，但事件标记为 DELETE。
4. NONE（无变化）
  - 何时使用：当新事实与现有记忆重复，或仅用词略有不同但传达完全相同的信息时。
  - 操作：不对现有记忆项做任何更改。
输出格式说明
你的最终输出必须是一个单独的 JSON 对象，键为 "memory"，包含一个记忆项列表。
列表中的每个项应包含：
- "id"：（字符串）标识符。对于 ADD，生成新 ID。对于所有其他操作，使用现有记忆中的旧 ID。
- "text"：（字符串）记忆项的最终文本。对于 DELETE，这将是原始文本。
- "event"：（字符串）"ADD"、"UPDATE"、"DELETE" 或 "NONE" 之一。
- "old_memory"：（字符串，可选）仅在 UPDATE 事件中包含此键。其值应为更新前的原始记忆文本。
示例应用

- 旧记忆：[{"id": "0", "text": "用户是一名软件工程师"}]

{
  "memory": [
    { "id": "0", "text": "用户是一名软件工程师", "event": "NONE" },
    { "id": "1", "text": "用户的名字是 John", "event": "ADD" }
  ]
}

3.设计的rubircs：
E1 抽取得到的事实列表是否覆盖本批用户发言中应被记住的关键可记忆要点（人物、事件、数字、计划等）？相对时间可在结合下方叙事时间理解后与 transcript 对照；明显遗漏重要要点判 0。
E2 抽取事实中是否无显著幻觉：不得出现 transcript 未支持的实体、因果、数字或事件？（有则 0）
E3 抽取中出现的相对时间（如「昨天」「上周」）在与 session 叙事时间 结合后，是否自洽、且不与 transcript 明显矛盾？若无任何相对时间表述则本条记 null。
E4 抽取是否避免过度合并导致在同一条事实中抹掉本可区分的独立要点（即信息粒度合理）？若本批仅一条极简发言、谈合并无意义则本条可记 null。

R1 当新事实与 prior 中任何一条记忆都无关时，本批输出是否以 ADD（或等价地仅新增、无错配 UPDATE）合理体现？
R2 当新事实是对某已有记忆的细节/语境增强时，是否使用 UPDATE（增强模式）？
R3 当新事实与某记忆同主题且为可合并的独立信息时，是否使用 UPDATE（综合模式）？
R4 当新事实与某记忆矛盾或使原信息过时，是否使用 DELETE（或合理删除标记）？
R5 当新事实与某记忆信息完全重复、仅措辞不同，是否使用 NONE 或不重复写入？
R6 增强模式下，更新后文本是否包含原记忆全部信息 + 新细节？
R7 综合模式下，更新后文本是否为一句通顺、符合中文表达的合并？
R8 DELETE 时是否仍保留 id、text、event（仅标记删除）？
