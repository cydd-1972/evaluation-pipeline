# 记忆写入 Rubric 与评测 Pipeline 项目记录

## 1. 项目整体目标

这个项目面向长程多轮对话中的记忆模型。完整链路包括：

1. **记忆写入（add / memory write）**：模型读入多轮会话，把未来可能有用的信息写成可复用记忆。
2. **记忆检索（search / retrieve）**：面对下游问题，从记忆或原始会话中找相关证据。
3. **问题回答（answer）**：基于检索结果回答 LoCoMo 风格问题。

当前重点是第一阶段：**训练记忆写入模型**。难点在于，记忆写入不像代码执行或数学题那样可以直接验证。下游 QA 分数虽然重要，但它是 `add → search → answer` 整条链路的结果，很难判断错误到底来自写入、检索还是回答。因此我们需要单独给写入阶段定义 **rubric reward**：直接评价最终写出的 memory 是否可靠、完整、精炼、结构化，并把这个分数作为训练或筛选写入模型的信号。

相关背景文档：

- `写入模型训练.md`
- `locomo数据清洗.md`

## 2. 为什么先做 conversation-level

记忆写入是多轮过程任务，可以有两类奖励：

- **PRM（过程奖励）**：对每一轮中间记忆打分。但中间状态不好不等于最终结果不好，因为后续轮次可能修复、合并或覆盖。
- **ORM（结果奖励）**：只对最终记忆打分。它忽略中间状态，但更接近最终目标，也更容易稳定实现。

对用户而言，每一轮都可能是最后一轮，中间结果也重要；但从训练和评测可操作性看，先做 conversation-level 最终记忆评分更稳。它相当于先建立一个写入结果的 ORM，然后后续再推进到 session-level 或更细粒度的过程奖励。

## 3. 数据与评测 pipeline

### 3.1 LoCoMo 数据清洗

LoCoMo 数据用于下游问答评测。清洗目标包括：

- 对齐 conversation / session 结构。
- 保留可追踪原始对话文本。
- 整理 QA 与 reference answer。
- 支持后续 `add → search → answer → score` 的完整 pipeline。

记录文件：

- `locomo数据清洗.md`

### 3.2 旧下游 QA pipeline

旧 pipeline 的核心流程是：

```text
conversation/session input
  → add memory
  → search relevant memory/context
  → answer question
  → compare with reference answer
  → llm_score / f1 / bleu
```

注意这里的分数是端到端 QA 分数，不是 memory rubric 分数。它反映整个链路，不只反映写入质量。

主要 workspace：

- `evaluation_pipeline/workspaces/`
- `evaluation_pipeline/v2_raw/workspaces/`
- `evaluation_pipeline/v4_global/workspaces/`
- `evaluation_pipeline/v5/workspaces/`

### 3.3 Rubric pipeline

rubric pipeline 直接输入：

- 完整原始 conversation。
- 最终 memory items。
- 固定 rubric 集合。

输出：

- 每条 rubric 的分数。
- 每个 conversation 的总分。
- 模型整体平均分。
- 日志与 summary 文件。

主要目录：

- `evaluation_pipeline/rubric/rubrics/`
- `evaluation_pipeline/rubric/prompts/`
- `evaluation_pipeline/rubric/outputs/`

## 4. Rubric 设计演进

### 4.1 初始 conversation-level rubric

相关文件：

- `evaluation_pipeline/rubric/rubrics/conversation_level_rubrics.json`
- `evaluation_pipeline/rubric/prompts/prompt_conversation_level.txt`
- `evaluation_pipeline/rubric/run_conversation_rubrics.py`

初始版本从 conversation-level 评价最终记忆，覆盖：

- sourceability / fabrication avoidance
- factual correctness
- temporal correctness
- atomicity
- nonredundancy
- coverage
- conciseness
- structural consistency

主要问题是太宽松。比如 “Does the final memory avoid fabrication?” 容易被模型理解成“整体看起来没有明显编造”，而不是逐条检查每条 memory 是否都有原文依据。再加上早期评分粒度偏粗，导致分数偏高、区分度不足。

### 4.2 strict v2 / v3：把 source 约束改成逐条检查

相关文件：

- `evaluation_pipeline/rubric/rubrics/conversation_level_rubrics_v2.json`
- `evaluation_pipeline/rubric/rubrics/conversation_level_rubrics_v3.json`
- `evaluation_pipeline/rubric/prompts/prompt_conversation_level_v2.txt`
- `evaluation_pipeline/rubric/prompts/prompt_conversation_level_v3.txt`
- `evaluation_pipeline/rubric/run_conversation_rubrics_v2.py`
- `evaluation_pipeline/rubric/run_conversation_rubrics_v3.py`

v2 / v3 的核心改进：

- 把泛泛的 sourceability 改成 **every-item sourceability**：每一条 memory 都必须能在原对话中找到可恢复来源。
- 把 factual / temporal 也改成逐条约束：不是整体不错就行，而是每条 memory 都不能事实错、时态错。
- 保留集合层面的 nonredundancy、coverage、conciseness、structural consistency。

这一步对应最关键的严格化建议：**不是问最终 memory 是否大体避免 fabrication，而是问每条 memory 是否都有 source**。

### 4.3 session delta / hybrid rubric：拆 precision 与 recall

相关文件：

- `evaluation_pipeline/rubric/rubrics/item_precision_rubrics.json`
- `evaluation_pipeline/rubric/rubrics/session_recall_rubrics.json`
- `evaluation_pipeline/rubric/rubrics/session_auxiliary_rubrics.json`
- `evaluation_pipeline/rubric/prompts/prompt_hybrid_delta_eval.txt`
- `evaluation_pipeline/rubric/run_hybrid_delta_rubrics.py`

这一版把写入质量拆成：

- **item precision**：写出来的 memory 是否正确、可追踪、不过度推断。
- **session recall**：当前 session 中重要信息是否被写入。
- **auxiliary quality**：结构、冗余、简洁等。

后来发现，如果 recall 依赖前置 fact 抽取，评测链路会变复杂，也会把抽取错误传给 rubric。最终决定：rubric 应该尽量不依赖额外代码或前置 fact extraction，而是直接基于原始对话和最终记忆判断。

### 4.4 original-text strict：直接用原文判断

相关文件：

- `evaluation_pipeline/rubric/rubrics/session_delta_rubrics_textcoverage.json`
- `evaluation_pipeline/rubric/prompts/prompt_session_delta_textcoverage.txt`
- `evaluation_pipeline/rubric/prompts/prompt_session_delta_originaltext_strict.txt`
- `evaluation_pipeline/rubric/run_session_delta_rubrics_textcoverage.py`
- `evaluation_pipeline/rubric/run_session_delta_originaltext_strict.py`

这一版的方向是回到原始文本：不先抽 salient facts，而是让 evaluator 直接比较原始 conversation 和 memory。这样更符合“写入结果是否由对话支持”的定义，也减少前置模块依赖。

### 4.5 fact / nonfact bucket rubric

相关数据：

- `evaluation_pipeline/datasets/add_snapshot_locomo.json`

这个文件中，每个 conversation 最后一个 session 里有多个 bucket：

- `fact`：add 阶段直接得到的事实记忆。
- 非 `fact`：基于 fact 总结出的更高层 bucket。

相关文件：

- `evaluation_pipeline/rubric/rubrics/conversation_fact_rubrics_continuous.json`
- `evaluation_pipeline/rubric/prompts/prompt_conversation_fact_continuous.txt`
- `evaluation_pipeline/rubric/rubrics/conversation_bucket_summary_rubrics.json`
- `evaluation_pipeline/rubric/prompts/prompt_conversation_bucket_summary_continuous.txt`
- `evaluation_pipeline/rubric/run_conversation_fact_and_summary_continuous.py`
- `evaluation_pipeline/rubric/run_conversation_bucket_summary_continuous.py`

这一步的重要变化：

- fact 与 nonfact 分开打分，因为二者目标不同。
- rubric 分数改成连续 `[0, 1]`。
- prompt 明确要求严格打分：`1.0` 代表完全没有明显问题，不能轻易给满分。
- fact 更强调 source、事实正确、时间正确；nonfact 更强调 schema fit、抽象质量、跨 fact 整合。

为了避免 rubric 完全依赖人工直觉，后续让 Gemini、DeepSeek、MiniMax 基于已有 fact rubric 生成更细粒度版本。

- `evaluation_pipeline/rubric/outputs/generate_finer_fact_rubrics_20260613_122735/deepseek_finer_fact_rubrics.json`

去重相关文件：

- `evaluation_pipeline/rubric/prompts/prompt_deduplicate_generated_fact_rubrics_conservative.txt`
- `evaluation_pipeline/rubric/prompts/prompt_deduplicate_generated_fact_rubrics_conservative_simplejson.txt`
- `evaluation_pipeline/rubric/run_deduplicate_generated_fact_rubrics_conservative.py`

去重原则是保守合并：

- 只有语义几乎完全一致、检查动作也一致时才合并。

- 措辞冲突时选更专业、简洁的表述。
- 严重性冲突时保留更严格的版本。

最后没有直接采用某个模型输出，而是人工整理成一版更可控的 rubric。

## 5. 当前打分方式

严格版 prompt：

- `evaluation_pipeline/rubric/prompts/prompt_consolidated_fact_rubrics_v1_strict.txt`

核心规则：

- 每条 rubric 输出连续分数 `[0, 1]`。
- `1.0` 表示该项基本完美，没有清楚问题。
- 有明确问题就不能给满分。
- 重复问题应低于 `0.8`。
- 系统性问题应更接近失败。
- 只根据原始对话和最终 fact memory 判断，不使用外部知识。

当前 prompt 要求 `conversation_score` 是 rubric 分数的未加权算术平均。脚本 summary 中仍保留旧版权重字段，但这些 key 没有完全映射到 26 个新 rubric id；如果后续要做真正加权，需要把权重配置更新成新 id。

## 9. MiniMax raw downstream 当前问题

尝试跑 `v2_raw + MiniMax downstream` 时，问题主要出现在 search 阶段：

- MiniMax 有输出，但经常不是合法 JSON。
- 常见错误包括自然语言分析、JSON 截断、字符串未闭合。
- 这不是“模型没有答案”，而是格式遵循失败。

当前原则：

- 不使用 fallback LLM，因为这会改变“MiniMax downstream”的实验定义。
- 可以使用 deterministic recovery，从 MiniMax 原始输出中规则提取合法 `conv*_session*` id。
- 对无法恢复的样本 skip，并记录 conversation id / qa id / skip 总数。
- 已解析成功的结果落盘到 `evaluation_pipeline/workspaces/matrix_raw_downstream_minimax_parallel/raw/llm/search_results.json`。

相关工程修改：

- `evaluation_pipeline/core/search/search_llm.py`
  - skip-on-failure
  - deterministic id recovery
  - chunked checkpoint
  - 修复 resumed records 导致进度双计数的问题
- `evaluation_pipeline/core/infra/checkpoint.py`
  - `search_skipped=true` 也视为该 search record 已处理，支持断点续跑
- `evaluation_pipeline/core/infra/llm_client.py`
  - 支持环境变量配置 API timeout、retry attempts、max tokens

注意：日志里的进度数字曾因 resumed records 双计数而偏大；真实进度应以 `search_results.json` 中已落盘 record 数为准。已经启动的进程不会自动加载代码修改，需要重启后才生效。

## 10. 当前结论

1. **只看 QA 分数不够**：QA 是端到端信号，不能稳定归因到写入阶段。
2. **conversation-level 是合理起点**：它比 session-level 更稳定，也比 QA 更直接评价写入结果。
3. **rubric 必须严格且细粒度**：早期 8 条太粗，容易高分；现在拆成 26 条后更适合诊断错误。
4. **连续分数优于 0/1**：连续 `[0, 1]` 更适合作为训练 reward。
5. **不应依赖前置 fact 抽取**：最终判断应直接基于原对话和最终记忆。
6. **fact 与 nonfact 应分开评**：fact 是写入阶段核心产物，summary / bucket 类内容目标不同。
7. **当前 rubric 分数仍偏高**：v5 fact 分数仍在 `0.84–0.86`，说明还需要人工校准和更强约束。

## 11. 后续建议

### 11.1 给 26 条 rubric 增加真实权重

高权重 / 可 hard cap：

- `explicit_textual_support`
- `entity_attribution_accuracy`
- `entity_attribute_accuracy`
- `relationship_accuracy`
- `event_detail_accuracy`
- `temporal_status_accuracy`
- `contradiction_avoidance`
- `negation_accuracy`

中权重：

- coverage 类
- duplicate 类
- `inference_boundary_compliance`

低权重：

- `conciseness`
- `structural_consistency`
- `field_naming_and_temporal_format_consistency`

### 11.2 加 hard cap

有些错误不应该只扣一点分，而应该限制最终分上限。例如：

- 如果很多 memory 没有 source，最终分不能高。
- 如果出现直接事实错误或实体错归因，最终分不能高。
- 如果 memory 内部有明显矛盾，最终分不能高。

说人话就是：**缺一点覆盖可以扣分，但编造、错归因、矛盾会让整份记忆不可信**。

### 11.3 做人工抽样校准

建议抽样若干 conversation：

- 人工标注每条 memory 是否有 source。
- 人工标注明显 hallucination、错归因、时态错误、重复和遗漏。
- 对比 evaluator 给分，校准 prompt 和权重。

这样可以判断当前 `0.84–0.86` 是真实质量高，还是 evaluator 仍然偏宽。

### 11.4 把 rubric reward 拆成训练信号

训练时不一定直接使用所有 rubric 的平均值，可以拆成：

- precision reward：写出的东西必须可靠。
- coverage reward：重要内容要写到。
- efficiency reward：避免冗余和低价值内容。
- structure reward：可检索、可更新、可维护。

其中 precision 类错误优先级最高，因为错误记忆通常比缺失记忆更危险。
